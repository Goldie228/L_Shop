## Кодекс поведения

### Наши обязательства

Мы обязуемся создавать приветливую и инклюзивную среду для всех участников проекта, независимо от опыта, пола, ориентации, инвалидности, внешности, расы, религии или национальности.

### Ожидаемое поведение

- Используйте приветливый и инклюзивный язык
- Уважайте различные точки зрения и опыт
- Принимайте конструктивную критику
- Фокусируйтесь на том, что лучше для сообщества
- Проявляйте эмпатию по отношению к другим участникам

### Недопустимое поведение

- Использование сексуализированного языка или образов
- Троллинг, оскорбительные комментарии и личные нападки
- Публичное или частное преследование
- Публикация личной информации других лиц без их согласия
- Иное поведение, которое может быть сочтено неуместным

## Как начать

### 1. Форк и клонирование

```bash
# Форкните репозиторий на GitHub
# Затем клонируйте свой форк
git clone https://github.com/YOUR_USERNAME/L_Shop.git
cd L_Shop

# Добавьте оригинальный репозиторий как upstream
git remote add upstream https://github.com/ORIGINAL_OWNER/L_Shop.git
```

### 2. Установка зависимостей

```bash
npm install
```

### 3. Настройка окружения

```bash
# Скопируйте пример конфигурации
cp .env.example .env

# Отредактируйте при необходимости
```

### 4. Проверка работоспособности

```bash
# Запустите тесты
npm test

# Запустите приложение
npm run dev
```

## Процесс разработки

### Git Workflow

#### Ветки

- `main` — защищённая ветка, изменения только через Pull Request
- `review` — пустая ветка для ревью (PR из main в review не закрывается)
- `feature/<модуль>-<имя>` — ветки разработчиков

#### Именование веток

```
feature/products-nikita    # Никита П. — продукты
feature/cart-timofey       # Тимофей — корзина
feature/orders-nikita-t    # Никита Т. — заказы
fix/auth-validation        # Исправление валидации
docs/api-update            # Обновление документации API
```

#### Создание ветки

```bash
# Синхронизируйтесь с upstream
git fetch upstream
git checkout main
git merge upstream/main

# Создайте ветку
git checkout -b feature/your-feature-name
```

### Формат коммитов

Используем **Conventional Commits** на английском языке:

```
<тип>(<область>): <описание>
```

#### Типы коммитов

| Тип | Описание | Пример |
|-----|----------|--------|
| `feat` | Новая функция | `feat(auth): add password reset` |
| `fix` | Исправление бага | `fix(cart): correct total calculation` |
| `docs` | Документация | `docs(api): update endpoints` |
| `style` | Форматирование | `style: fix indentation` |
| `refactor` | Рефакторинг | `refactor(user): extract validation` |
| `test` | Тесты | `test(auth): add login tests` |
| `chore` | Обслуживание | `chore: update dependencies` |

#### Правила

1. **Единица изменений** — один коммит, одно логическое изменение
2. **Описание** — на русском языке, в повествовательном наклонении
3. **Длина** — не более 72 символов в заголовке

```bash
# ПЛОХО
git commit -m "fix"
git commit -m "исправил баги"
git commit -m "feat: добавил новую функциональность которая позволяет пользователям..."

# ХОРОШО
git commit -m "feat(auth): add session expiration check"
git commit -m "fix(cart): correct item quantity validation"
git commit -m "docs: add development guide"
```

### Pull Request

#### Создание PR

1. Убедитесь, что ваш код проходит все тесты
2. Убедитесь, что код соответствует стандартам (lint)
3. Запушьте ветку в свой форк
4. Откройте Pull Request в `main` оригинального репозитория

#### Шаблон описания PR

```markdown
## Описание

Краткое описание изменений.

## Тип изменений

- [ ] Bug fix (исправление бага)
- [ ] New feature (новая функция)
- [ ] Breaking change (изменение, нарушающее обратную совместимость)
- [ ] Documentation update (обновление документации)

## Как тестировалось

Описание того, как тестировались изменения.

## Чеклист

- [ ] Код следует стайл-гайду проекта
- [ ] Добавлены тесты для новых функций
- [ ] Обновлена документация
- [ ] Все тесты проходят
```

#### Code Review

1. Получите минимум 2 аппрува от коллег
2. Исправьте все замечания
3. Тимлид (Глеб) делает финальный merge

#### После одобрения

```bash
# Синхронизируйтесь с upstream
git fetch upstream
git checkout main
git merge upstream/main

# Удалите слитую ветку
git branch -d feature/your-feature-name
git push origin --delete feature/your-feature-name
```

## Стандарты кода

### TypeScript

- **Строгая типизация**: `strict: true`, `noImplicitAny: true`
- **Запрет `any`**: используйте `unknown` с проверками
- **Импорты**: сортировка по eslint-plugin-import

### Именование

| Элемент | Стиль | Пример |
|---------|-------|--------|
| Переменные | camelCase | `userName`, `isLoggedIn` |
| Функции | camelCase | `getUserById`, `calculateTotal` |
| Классы | PascalCase | `UserService`, `AuthMiddleware` |
| Интерфейсы | PascalCase | `User`, `CartItem` (без префикса I) |
| Константы | UPPER_SNAKE_CASE | `SESSION_DURATION_MINUTES` |
| Файлы | kebab-case | `user.service.ts`, `auth.middleware.ts` |

### Форматирование

- **Отступы**: 2 пробела
- **Кавычки**: одинарные
- **Длина строки**: максимум 100 символов
- **Скобки**: открывающая на той же строке

### ESLint + Prettier

Конфигурация:
- `airbnb-typescript/base`
- Single quotes
- Trailing commas
- Max line width: 100

```bash
# Проверка кода
npm run lint

# Автоматическое исправление
npm run lint -- --fix

# Форматирование
npm run format
```

## Тестирование

### Обязательные требования

- Минимальное покрытие: **80%**
- Новые функции должны иметь тесты
- Исправление багов должно включать регрессионные тесты

### Unit тесты (Jest)

```bash
# Все тесты
npm test

# Конкретный файл
npm test -- user.service.test.ts

# Режим слежения
npm run test:watch

# Отчёт о покрытии
npm run test:coverage
```

### Структура теста (AAA)

```typescript
describe('UserService', () => {
  describe('createUser', () => {
    it('должен создать пользователя с корректными данными', () => {
      // Arrange
      const userData = {
        name: 'Иван',
        email: 'ivan@test.com',
        password: 'password123',
      };

      // Act
      const user = createUser(userData);

      // Assert
      expect(user).toBeDefined();
      expect(user.name).toBe('Иван');
    });
  });
});
```

### E2E тесты (Cypress)

```bash
# Headless режим
npm run test:e2e

# Интерактивный режим
npm run test:e2e:open
```

### Data-атрибуты

Для E2E тестирования используйте специальные data-атрибуты:

| Атрибут | Элемент |
|---------|---------|
| `data-registration` | Форма регистрации |
| `data-delivery` | Форма доставки |
| `data-title` | Название товара на главной |
| `data-price` | Цена товара на главной |
| `data-title="basket"` | Название товара в корзине |
| `data-price="basket"` | Цена товара в корзине |

## Отчёт об ошибках

### Перед созданием issue

1. Проверьте, что баг ещё не зарегистрирован
2. Убедитесь, что используете последнюю версию
3. Попробуйте воспроизвести баг на чистой установке

### Шаблон баг-репорта

```markdown
**Описание бага**
Чёткое описание того, что происходит.

**Шаги для воспроизведения**
1. Перейдите на '...'
2. Нажмите на '...'
3. Прокрутите до '...'
4. Увидьте ошибку

**Ожидаемое поведение**
Чёткое описание того, что должно происходить.

**Скриншоты**
Если применимо, добавьте скриншоты.

**Окружение**
- OS: [например, macOS, Windows, Linux]
- Browser: [например, Chrome, Firefox]
- Version: [например, 22]
- Node.js: [например, v20.10.0]

**Дополнительная информация**
Любая другая информация о проблеме.
```

## Чеклист для PR

### Перед отправкой на review

- [ ] Код проходит `npm run lint` без ошибок
- [ ] Код отформатирован `npm run format`
- [ ] Все тесты проходят `npm test`
- [ ] Нет `any` типов
- [ ] Все функции типизированы
- [ ] Добавлены комментарии к сложной логике
- [ ] Обновлена документация (если нужно)
- [ ] Добавлены тесты для нового кода
- [ ] Data-атрибуты на месте (для UI компонентов)

### При просмотре кода

- [ ] Код читается легко
- [ ] Имена переменных и функций понятны
- [ ] Нет дублирования кода
- [ ] Обработаны все возможные ошибки
- [ ] Нет утечек памяти (в async функциях)
- [ ] Логика разбита на функции
- [ ] Тесты покрывают новый код
- [ ] Нет hardcoded значений (используйте константы)

### После review

- [ ] Все замечания исправлены
- [ ] Проведена самопроверка
- [ ] Коммиты соответствуют conventional commits
